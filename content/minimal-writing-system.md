## 工具的陷阱

作为一个写作者，我尝试过很多写作工具：从功能丰富的 Word 到简洁的 Markdown 编辑器，从在线协作平台到本地知识管理软件。每次尝试新工具时，我都会花费大量时间去学习它的功能、配置它的环境、优化它的设置。

但最终我发现，**工具越复杂，创作的阻力就越大**。

## 回归本质

写作的本质是什么？是把想法转化为文字。仅此而已。

一个好的写作系统应该满足以下条件：

- **启动迅速** - 打开就能写，没有加载时间
- **格式简单** - 不需要纠结排版，专注于内容
- **易于检索** - 能够快速找到之前的想法
- **版本可控** - 不用担心丢失或覆盖
- **发布方便** - 写完就能分享

## 我的极简方案

### Markdown + Git

使用 Markdown 格式存储所有文章，用 Git 进行版本管理。这套组合的优势：

1. **纯文本** - 永远不会被锁定在某个特定软件中
2. **版本历史** - 每一次修改都有记录，可以随时回溯
3. ** diff 友好** - 清楚看到每次改了什么
4. **协作简单** - 多人编辑没有冲突
5. **发布灵活** - 可以轻松转换为 HTML、PDF 等各种格式

### 文件组织结构

```
content/
├── 2024/
│   ├── thinking/
│   ├── tech/
│   └── life/
└── drafts/
```

按年份和主题分类，草稿单独存放。保持扁平，不要嵌套太深。

## 写作流程

1. **随时记录** - 想法出现时立即在草稿区记下
2. **定期整理** - 每周回顾草稿，扩展成完整文章
3. **专注写作** - 使用全屏编辑器，关闭所有通知
4. **完成发布** - 移动到正式目录，提交 Git，自动部署

## 技术实现

博客使用 Next.js 构建，通过 GitHub Actions 自动部署到 GitHub Pages。整个流程完全自动化：

```bash
# 本地写作
echo "# 新文章" > content/new-post.md
git add . && git commit -m "add new post"
git push

# 自动部署到线上
```

不需要登录任何后台，不需要点击发布按钮。**Git push 即发布**。

## 结语

最好的工具是那个让你忘记它存在的工具。当技术细节不再分散注意力，我们才能专注于真正重要的事情：**写出有价值的文字**。
